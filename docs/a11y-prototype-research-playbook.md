# A11y Prototype Research Playbook

프로토타입 단계에서 접근성 리서치를 **가볍게 시작**하기 위한 실행용 플레이북입니다.

## 1) 언제 이 플레이북을 쓰나

- Figma만으로는 실제 보조기기/키보드 사용성을 확인하기 어려울 때
- 출시 직전이 아니라, 요구정의/프로토타입 단계에서 위험을 먼저 줄이고 싶을 때
- 팀이 작아서 복잡한 리서치 프로세스 대신 **작고 반복 가능한 루틴**이 필요할 때

---

## 2) 30~60분 실행 루틴

### Step 0. 사전검증 (5분)

목표: 리서치 시작 전에 프로토타입의 기본 접근성 준비 상태를 빠르게 확인

- [ ] 읽기 순서(제목→본문→주요 행동)가 시각 흐름과 논리 흐름에 맞는지 확인했다
- [ ] 주요 입력/버튼의 라벨이 명확하고 중복 의미가 없는지 확인했다
- [ ] 핵심 이미지/아이콘의 `alt` 또는 대체 텍스트 정책을 정했다
- [ ] 키보드 흐름(Tab/Shift+Tab, 모달 진입·종료 후 포커스 복귀)을 한 번 점검했다
- [ ] 스크린리더용 요소 이름(heading/button/field)이 실제 UI 문맥과 일치하는지 확인했다

빠른 기록 포맷(회차 시작 전 1분):
- 읽기 순서: 통과 / 보완 필요
- 라벨: 통과 / 보완 필요
- alt: 통과 / 보완 필요
- 키보드: 통과 / 보완 필요

### Step A. Survey (10~15분)

목표: 이해도/혼란 지점을 빠르게 수집

- [ ] 핵심 과업 1개를 짧게 설명했다
- [ ] 상태 전달(색만 vs 텍스트+아이콘) 이해 여부를 물었다
- [ ] 확대(200%)/다크모드에서 읽기 어려운 지점을 물었다

### Step B. Co-design (15~20분)

목표: 사용자와 함께 흐름/문구를 정리

- [ ] 핵심 플로우 1개(예: 검색→상세→제출)를 선정했다
- [ ] 버튼/입력 라벨 문구를 함께 정리했다
- [ ] 탭 순서(첫 포커스, 모달 닫힘 후 복귀)를 합의했다

### Step C. Rapid HTML (15~25분)

목표: 브라우저에서 키보드/포커스 동작을 현실적으로 검증

- [ ] 테스트 페이지에서 Tab/Shift+Tab 이동을 확인했다
- [ ] `:focus-visible` 가시성을 확인했다
- [ ] `label`/`aria-label` 접근성 이름을 확인했다
- [ ] 200% 확대 시 잘림/겹침이 없는지 확인했다

권장 파일: `docs/a11y-rapid-harness.html`

### Step D. Wizard-of-Oz (10~15분, 선택)

목표: 아직 구현되지 않은 흐름도 “사람 중재”로 리서치 공백 없이 검증

- [ ] 참가자에게 프로토타입 한계를 먼저 안내했다
- [ ] 진행자가 시스템 응답(로딩/결과/오류)을 수동으로 제공했다
- [ ] 막힌 지점(문구/순서/피드백 부족)을 기록했다
- [ ] 실제 구현 전환 시 필요한 최소 UI 상태를 정리했다

### 대체 리서치 실행 체크리스트 (회차 종료 전)

- [ ] Survey / Co-design / Rapid HTML / Wizard-of-Oz 중 **최소 2개 방식**을 실행했다
- [ ] 각 방식별로 “막힘 1개 + 개선안 1개”를 기록했다
- [ ] 다음 회차에서 재검증할 항목을 1~2개로 압축했다

### 60초 방식 선택 가이드 (가볍게)

시간과 상황에 따라 아래처럼 시작합니다.

- **10~15분만 가능**: Survey + Rapid HTML(핵심 1화면)
- **라벨/문구 충돌이 많음**: Co-design + Rapid HTML
- **구현 전 컨셉 검증이 목적**: Survey + Wizard-of-Oz
- **키보드/포커스 이슈 재현이 목적**: Rapid HTML 우선 + Survey 보조

빠른 결정 규칙:
- [ ] 목표가 “이해도”인지 “조작성”인지 먼저 고른다
- [ ] 시간 예산에 맞춰 2개 방식만 선택한다
- [ ] 다음 회차로 넘길 검증 항목 1개를 미리 적어둔다

## 4) 운영 팁 (가볍게 유지)

- 한 번에 모든 화면을 하지 말고, **핵심 플로우 1개**만 선택
- 회차마다 이슈를 1~2개만 확실히 줄이는 것을 목표로 설정
- 결과는 PR/WORKLOG에 남겨 다음 회차에서 이어서 개선
