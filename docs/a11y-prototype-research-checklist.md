# A11y Prototype Research Checklist (Alternative Methods)

프로토타입 단계에서 Figma만으로 접근성 검증이 어려울 때, 아래 **대체 방법**으로 빠르게 리스크를 줄일 수 있습니다.

## 1) 방법 선택 (상황별)

- **설문(Survey):** 빠르게 인지/이해도 확인이 필요할 때
- **코디자인(Co-design):** 사용자와 함께 플로우를 재설계해야 할 때
- **Rapid HTML 테스트:** 키보드/포커스/확대/스크린리더 기초 동작을 실제 브라우저에서 확인할 때

---

## 2) 실행 체크리스트

### A. 설문 (Survey)
- [ ] 과업 시나리오 1~2개를 문장으로 제시했다
- [ ] 상태 전달(색만/아이콘+텍스트) 이해 여부를 물었다
- [ ] 확대(200%) 또는 다크모드에서 읽기 어려운 지점을 수집했다
- [ ] 자유서술 응답(불편 포인트)을 1개 이상 받았다

### B. 코디자인 (Co-design)
- [ ] 핵심 플로우(예: 검색→상세→제출) 1개를 선정했다
- [ ] 버튼 이름/입력 라벨 문구를 사용자와 같이 정리했다
- [ ] 키보드 탭 순서(처음 포커스/모달 닫힘 후 복귀)를 합의했다
- [ ] 결과를 다음 프로토타입/이슈에 반영할 액션으로 기록했다

### C. Rapid HTML 테스트
- [ ] 테스트 페이지(예: `docs/a11y-rapid-harness.html`)를 준비했다
- [ ] Tab/Shift+Tab으로 모든 인터랙션 요소에 도달 가능하다
- [ ] `:focus-visible` 표시가 명확하다(경계/대비)
- [ ] 버튼/입력의 visible label과 접근성 이름(`label`/`aria-label`)이 목적과 일치한다
- [ ] 브라우저 200% 확대에서도 내용 잘림/겹침이 없다

---

## 3) PR/이슈 템플릿 (복붙용)

```markdown
### Prototype A11y Research (Alternative Methods)
- [ ] Survey 수행
- [ ] Co-design 세션 수행
- [ ] Rapid HTML 테스트 수행

### Findings
- 핵심 문제 1:
- 핵심 문제 2:

### Next Actions
- [ ] 다음 스프린트 반영 항목 1개 이상 정의
```

## 4) 커스터마이징 포인트

- 체크리스트 항목은 팀/도메인(공공/금융/커머스)에 맞게 항목 추가 가능
- Rapid HTML 파일은 컴포넌트(버튼/모달/폼) 1~2개만 넣어 가볍게 유지 권장
- 자동화가 부담되면 수동 체크 + WORKLOG 기록만으로도 회귀 리스크를 줄일 수 있음
